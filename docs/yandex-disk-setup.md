# Настройка бэкенда Яндекс.Диск

Инструкция по использованию **Яндекс.Диска** как хранилища для зашифрованного vault
(альтернатива WebDAV). Проведёт от создания приложения в Яндексе до настройки encryptor и плагина.

> **Почему REST, а не WebDAV.** На **бесплатном тарифе** Яндекс.Диска запись по WebDAV отключена —
> сервер отвечает `402 Payment Required` («WebDAV is not available for the free tariff»).
> А **REST API работает и на бесплатном тарифе**, поэтому плагин и encryptor ходят именно через REST.

## 1. Создать приложение в Яндекс OAuth

Токен для доступа к Диску выдаётся зарегистрированному OAuth-приложению. Домен и свой сервер **не нужны**.

1. Открой **<https://oauth.yandex.ru/>** и залогинься под своим Яндекс-аккаунтом.
2. Нажми **«Создать приложение»**.
3. ⚠️ **Ключевой момент.** В окне выбора типа выбери **«Для доступа к API или отладки»**.
   - НЕ выбирай вариант с платформами (Web-сервисы / iOS / Android) — это «приложение для входа»,
     в нём **нет** прав Яндекс.Диска, и ты застрянешь на экране с личными данными (имя/почта).
   - Если тебя всё равно перекинуло на мастер с «Platforms» — вернись и выбери тип «Для доступа к API»,
     либо открой напрямую **<https://oauth.yandex.ru/client/new/>**.
4. Заполни:
   - **Название сервиса** — любое, например `obsidian-sync`.
   - **Контактная почта** — свой email.
5. В секции **«Доступ к данным»** добавь права **Яндекс.Диск REST API**:
   - **`cloud_api:disk.read`** — «Чтение всего Диска»;
   - **`cloud_api:disk.write`** — «Запись в любом месте на Диске».
6. **Redirect URI** для приложений API-доступа фиксированный — **`https://oauth.yandex.ru/verification_code`**
   (менять не нужно, он подставляется сам; именно он нужен для получения токена через браузер).
7. Нажми **«Создать приложение»**. Скопируй со страницы приложения **Client ID**
   (Client secret для нашего сценария не понадобится).

Документация Яндекса: [Регистрация приложения для доступа к API](https://yandex.com/dev/id/doc/ru/register-api) ·
[Доступ к API Яндекс.Диска](https://yandex.ru/dev/disk/api/concepts/quickstart.html).

## 2. Получить OAuth-токен

Самый простой способ — **implicit-flow** прямо в браузере (без своего сервера):

1. Открой в браузере ссылку, подставив свой `Client ID`:
   ```
   https://oauth.yandex.ru/authorize?response_type=token&client_id=ВАШ_CLIENT_ID
   ```
2. Подтверди доступ. Тебя перекинет на страницу `oauth.yandex.ru/verification_code`, а в **адресной строке**
   после `#` будет токен:
   ```
   https://oauth.yandex.ru/verification_code#access_token=y0__ВОТ_ЭТО_ЗНАЧЕНИЕ&token_type=bearer&expires_in=31536000
   ```
3. Скопируй значение **`access_token`** (длинная строка после `access_token=` и до `&`).

Проверить токен (должно вернуть `200`):
```bash
curl -s -o /dev/null -w "%{http_code}\n" \
  -H "Authorization: OAuth ВАШ_ТОКЕН" https://cloud-api.yandex.net/v1/disk/
```

> Токен живёт около года. Когда истечёт, запросы начнут отдавать `401` — просто повтори шаг 2 и получи новый.
> Автоматическое обновление токена на headless-сервере (device-flow `--login`) пока не реализовано.

## 3. Настроить encryptor (сервер/VPS)

Encryptor шифрует исходный vault и заливает блобы + `manifest.enc` на Диск в папку `disk:/<remoteBase>/`.
Папка создаётся автоматически.

`config.json`:
```json
{
  "backend": "yandex",
  "yandexToken": "ВАШ_ТОКЕН",
  "remoteBase": "second-brain",
  "passphrase": "ВАША_ПАРОЛЬНАЯ_ФРАЗА",
  "sourceDir": "/абсолютный/путь/к/vault",
  "statePath": "/абсолютный/путь/к/state.json"
}
```

Запуск:
```bash
node encryptor.mjs --config /path/to/config.json          # инкрементальный синк
node encryptor.mjs --config /path/to/config.json --full   # перешифровать всё заново
```

Секреты удобнее держать в переменных окружения (тогда `yandexToken`/`passphrase` можно не писать в файл):
```bash
YANDEX_TOKEN='…' PASSPHRASE='…' node encryptor.mjs --config /path/to/config.json
```
Env-переменные для Яндекса: `BACKEND=yandex`, `YANDEX_TOKEN`, `REMOTE_BASE` (плюс общие `PASSPHRASE`, `SOURCE_DIR`, `STATE_PATH`).

Полный справочник по encryptor (все ключи, флаги, коды выхода, установка одной командой):
**[encryptor/README.md](../encryptor/README.md)**.

## 4. Настроить плагин (Obsidian)

В настройках плагина **WebDAV Decrypt Sync**:

| Поле | Значение |
|---|---|
| **Backend** | `Yandex.Disk` |
| **Yandex.Disk OAuth token** | твой `access_token` из шага 2 |
| **Remote base** | папка на Диске — та же, что `remoteBase` у encryptor (напр. `second-brain`) |
| **Passphrase** | **та же** пассфраза, что у encryptor |
| **Target folder** | куда класть в vault (пусто = корень) |
| **Yandex.Disk API endpoint** | по умолчанию `https://cloud-api.yandex.net/v1/disk`; меняй, только если ходишь через прокси/зеркало API |

Затем:
1. **Test connection** → должно быть `✓ Connection OK`.
2. **Sync now** (кнопка/ribbon/команда) → файлы скачаются и расшифруются.
3. По желанию выставь **Sync interval** и **Sync on open**.

> **iOS:** фоновой синхронизации нет — синк идёт только пока приложение открыто. Локальный vault
> трактуется как read-only: правки на телефоне перезаписываются при следующем синке.

## 5. Обслуживание и безопасность

- **`passphrase` должна совпадать** в encryptor и в плагине, иначе расшифровка провалится
  (несовпадение GCM-тега манифеста → синк прервётся до записи).
- **Токен = полный доступ к Диску.** Не коммить его в репозиторий, передавай через env. Скомпрометированный
  токен отзывается на странице приложения в [oauth.yandex.ru](https://oauth.yandex.ru/) (или удалением приложения).
- **Смена пассфразы:** нужно перешифровать всё (`encryptor … --full`) и сбросить локальный стейт в плагине
  командой **Reset local sync state**.
- **Особенность REST:** на каждый файл — два запроса (получить одноразовую ссылку, затем передача),
  так что раундтрипов больше, чем у WebDAV; на больших первичных синках это заметно.

## 6. Диагностика

| Симптом | Причина / решение |
|---|---|
| На экране создания приложения только имя/почта, нет прав Диска | Выбран тип «приложение для входа». Создай заново с типом **«Для доступа к API или отладки»** (см. §1). |
| `GET … 401` | Токен истёк или отозван — получи новый (§2). |
| `GET manifest.enc failed: 404` в плагине | Неверный **Remote base** (папка на Диске) — он должен совпадать с `remoteBase` encryptor. |
| `402 Payment Required` | Это WebDAV на бесплатном тарифе. Используй бэкенд **yandex** (REST), а не webdav. |
| «Decryption failed» / синк падает до записи | Пассфразы encryptor и плагина не совпадают. |
